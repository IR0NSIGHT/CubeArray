# Block Model JSON Format

Reference notes on the vanilla Minecraft block model JSON format, as shipped in resource packs
(e.g. `assets/minecraft/models/block/*.json` inside the "Pixel Perfection Legacy" texture pack
bundled in `src/main/resources/textures/`). This is the format CubeArray's `SubBlock`/`FaceTexture`
data model (see `src/main/java/org/ironsight/CubeArray/`) is meant to mirror.

## Example

`assets/minecraft/models/block/acacia_fence_post.json`:

```json
{
  "credit": "Made with Blockbench",
  "textures": {
    "side": "block/acacia_fence_side",
    "top": "block/acacia_fence_top",
    "particle": "block/acacia_planks"
  },
  "elements": [
    {
      "name": "Center post",
      "from": [6, 0, 6],
      "to": [10, 16, 10],
      "faces": {
        "north": {"uv": [6, 0, 10, 16], "texture": "#side"},
        "east":  {"uv": [6, 0, 10, 16], "texture": "#side"},
        "south": {"uv": [6, 0, 10, 16], "texture": "#side"},
        "west":  {"uv": [6, 0, 10, 16], "texture": "#side"},
        "up":    {"uv": [6, 6, 10, 10], "texture": "#top", "cullface": "up"},
        "down":  {"uv": [6, 6, 10, 10], "texture": "#top", "cullface": "down"}
      }
    }
  ]
}
```

## Top-level fields

| Field | Type | Meaning |
|---|---|---|
| `parent` | string | Path to another model this one inherits from (e.g. `"block/cube_all"`, `"builtin/generated"`). Child `textures`/`elements` override or fill in `#`-referenced slots from the parent. Not present when a model is fully self-contained (as in the example above). |
| `textures` | object | Maps a **texture key** (arbitrary name, e.g. `"side"`, `"top"`, `"particle"`, `"all"`, `"end"`) to either a resolved sprite path (`"block/acacia_fence_side"`) or another indirection `"#otherKey"` that gets resolved against a parent model. `"particle"` is the sprite used for break/step particles, not rendered on any face. |
| `elements` | array | List of cuboids (see below). Each is independently positioned/textured; a model can have any number (e.g. fence models split into a post + arm bars). Absent for models that only set `textures` and rely on a `parent` like `"builtin/generated"` (item sprites) to supply geometry. |
| `ambientocclusion` | bool | Whether to bake AO for this model. Rarely relevant for a viewer without full lighting. |
| `display` | object | Per-context (`"gui"`, `"thirdperson_righthand"`, `"firstperson_lefthand"`, etc.) transforms for item rendering. Not relevant to CubeArray, which only renders placed blocks. |

## `elements[]` â€” one cuboid

| Field | Type | Meaning |
|---|---|---|
| `from` / `to` | `[x,y,z]` | Opposite corners of the cuboid, in a **[0,16] per-axis coordinate space** where 16 units = one full block. E.g. `"from": [6,0,6], "to": [10,16,10]` is a 4Ă—16Ă—4 post centered on x/z. This maps directly to `SubBlock.from`/`SubBlock.to`. |
| `rotation` | object | `{"origin": [x,y,z], "axis": "x"\|"y"\|"z", "angle": degrees, "rescale": bool}`. Rotates the whole element around an axis-aligned pivot. **Not currently modeled** in `SubBlock` â€” CubeArray applies rotation at the whole-block level instead (`SchemReader`'s `rotationPalette`), not per-element. |
| `shade` | bool | Whether the element receives directional shading. Default `true`. Not modeled. |
| `faces` | object | Map from face name (`"north"`, `"south"`, `"east"`, `"west"`, `"up"`, `"down"`) to a face definition (below). A face absent from the map is not rendered at all (e.g. an internal face permanently hidden by geometry). This maps to `SubBlock.faces` (keyed by the `Face` enum). |

### Face names and directions

Face names are absolute cardinal directions in the model's own local space (not relative to
block rotation/blockstate â€” blockstate-driven rotation, e.g. a stair's `facing` property, is
applied afterward by the game/renderer, rotating the whole baked model including its face
assignments):

- `north` = -Z, `south` = +Z, `east` = +X, `west` = -X, `up` = +Y, `down` = -Y

## `faces.<name>` â€” one face definition

| Field | Type | Meaning |
|---|---|---|
| `texture` | string | A `"#key"` reference into the model's (or an inherited) `textures` map. Resolves to a sprite path like `"block/acacia_fence_side"`. Maps to `FaceTexture.texture`, but **already resolved** â€” CubeArray's `FaceTexture.texture` should hold the final sprite path, not the raw `"#side"` reference, since a `SubBlock` shouldn't need the enclosing model's `textures` table to be meaningful. |
| `uv` | `[u1,v1,u2,v2]` | Texture-space rectangle to sample, in **[0,16] units relative to the referenced sprite** (i.e. same 0-16 space as element coordinates, independent of the sprite's actual pixel resolution). If omitted, UV is auto-generated by projecting the element's own position onto that face. Maps to `FaceTexture.uv`. |
| `cullface` | string | One of the six face names. If the neighboring block in that direction is a full opaque face, this face is culled (not rendered) for performance. Maps to `FaceTexture.cullface` (nullable â€” `null` means never culled / always render, which is the safe default for a viewer with irregular sub-block geometry where "neighbor" isn't well-defined per sub-block anyway). |
| `rotation` | int | One of `0`, `90`, `180`, `270`. Rotates the UV rectangle on the sprite (e.g. so a single log-side texture can be reused rotated for horizontal logs). **Not currently modeled.** |
| `tintindex` | int | Non-negative index selecting a tint color (e.g. grass/leaves biome coloring) applied by the game at bake time. **Not currently modeled** â€” CubeArray already has its own coarser per-block-type `colorPalette` in `SchemReader.CubeSetup` that serves a similar purpose today. |

## Multi-element models

A model can list several `elements`, each with its own geometry and face textures, rendered
together as one block. This is how vanilla represents non-cubic blocks:

- **Fences**: split into a `*_post.json` (single centered post, 4Ă—16Ă—4) and a `*_side.json`
  (two horizontal bars reaching from the block edge toward the post), one element per bar. See
  `SchemReader.FENCE_POST_MODEL` / `FENCE_ARM_MODEL` for CubeArray's hardcoded equivalents.
- **Stairs**: a full slab element plus a smaller step element.
- **Fences/walls/stairs generally**: connections to neighboring blocks (north/east/south/west
  booleans in the blockstate) select *which* variant model â€” and thus which elements â€” get baked
  and placed, rather than the model itself branching at render time.

## What CubeArray currently models vs. not

Implemented (`Face.java`, `FaceTexture.java`, `SubBlock.java`):
- `from` / `to` per element
- one `FaceTexture` (resolved `texture` sprite name + `uv` rect + optional `cullface`) per `Face`,
  stored as `SubBlock.faces: Map<Face, FaceTexture>`

Not yet modeled (all real vanilla-format features, listed for when they become needed):
- element `rotation` (per-cuboid rotation/pivot) â€” CubeArray currently only rotates whole blocks
- `tintindex` / biome tinting
- face UV `rotation`
- `parent` model inheritance and `textures` indirection resolution (the `"#key"` lookup itself â€”
  by the time a `FaceTexture` exists, this should already be resolved)
- `display` transforms (item rendering â€” out of scope, CubeArray only renders placed blocks)
- auto-generated UV when a face's `uv` is omitted
